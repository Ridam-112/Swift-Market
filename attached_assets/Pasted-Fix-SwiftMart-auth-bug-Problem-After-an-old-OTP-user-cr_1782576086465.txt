Fix SwiftMart auth bug:

Problem:
After an old OTP user creates a password once, next login still asks to create password again.

Expected:
- If user already created password, next time mobile number দিলে direct password login screen দেখাবে.
- Create password screen should show only when password_hash/passwordHash is NULL or empty.

Debug and fix:

1. Check /api/auth/set-password route:
- After user creates password, bcrypt hash must be saved into users.password_hash.
- Make sure the update query uses the correct DB column name:
  password_hash in Neon / passwordHash in app mapping.
- After update, verify affected row count is 1.
- Return hasPassword: true in response.

2. Check /api/auth/check-phone route:
- It must check real database value:
  password_hash IS NOT NULL AND password_hash != ''
- If password_hash exists, return:
  { exists: true, hasPassword: true }
- If missing, return:
  { exists: true, hasPassword: false }

3. Check frontend:
- After successful set-password, clear old check-phone state.
- Do not keep hasPassword: false in React state/localStorage.
- Redirect user to logged-in app OR password login screen.
- On next mobile number submit, call /api/auth/check-phone again fresh.
- Do not show create password screen if API returns hasPassword: true.

4. Check naming mismatch:
- Neon column may be password_hash.
- Drizzle/schema field may be passwordHash.
- Ensure both set-password and check-phone use the same mapped field correctly.

5. Add server logs temporarily:
- Log user id/mobile number
- Log whether password_hash exists before set-password
- Log whether password_hash exists after set-password
- Log check-phone hasPassword result

6. Important:
- Do not reset Neon database.
- Do not delete users/products/orders/vendors.
- Do not recreate users table.
- Only fix auth logic.

Test:
- Pick one old OTP user.
- Run check-phone → hasPassword false.
- Set password.
- Run check-phone again → must return hasPassword true.
- Login with mobile + password → must work.
- Logout.
- Enter same mobile again → must show password login, NOT create password.
